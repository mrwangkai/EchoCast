6. Passed player as dependency to stabilize @ObservedObject
7. Finally lifted the sheet presentation to ContentView level (successful fix)

#### Final Solution: Lift sheet to ContentView level (48e9ca1) ✅

**Root Cause:** The `.tabViewBottomAccessory` closure re-evaluates when any @Published property changes (like `player.isPlaying`). This causes MiniPlayerBar to reconstruct, which invalidates its local `@State` (including `activeSheet`), causing the sheet to dismiss.

**Solution:**
- Added `@State private var showMiniPlayerNoteSheet = false` to ContentView
- Added `@Binding var showNoteSheet: Bool` to MiniPlayerBar
- Removed `enum MiniPlayerSheet` and `@State private var activeSheet` from MiniPlayerBar
- Removed `.sheet(item: $activeSheet)` modifier from MiniPlayerBar
- Changed add note button action to `showNoteSheet = true`
- Added `.sheet(isPresented: $showMiniPlayerNoteSheet)` at ContentView level
- Sheet now presented from stable ContentView scope, not re-evaluated `.tabViewBottomAccessory` closure

**Code Changes:**

**ContentView.swift:**
```swift
@State private var showMiniPlayerNoteSheet = false

.tabViewBottomAccessory(isEnabled: player.currentEpisode != nil) {
    if let episode = player.currentEpisode, let podcast = player.currentPodcast {
        MiniPlayerBar(
            episode: episode,
            podcast: podcast,
            showFullPlayer: $showFullPlayer,
            showNoteSheet: $showMiniPlayerNoteSheet,  // ← New binding
            player: player,
            namespace: playerAnimation
        )
    }
}

.sheet(isPresented: $showMiniPlayerNoteSheet) {
    if let episode = player.currentEpisode, let podcast = player.currentPodcast {
        NoteCaptureSheetWrapper(
            episode: episode,
            podcast: podcast,
            currentTime: player.currentTime
        )
        .environment(\.managedObjectContext, viewContext)
    }
}
```

**MiniPlayerBar:**
```swift
struct MiniPlayerBar: View {
    let episode: RSSEpisode
    let podcast: PodcastEntity
    @Binding var showFullPlayer: Bool
    @Binding var showNoteSheet: Bool  // ← New binding
    @ObservedObject var player: GlobalPlayerManager
    var namespace: Namespace.ID

    // MiniPlayerSheet enum removed
    // @State activeSheet removed

    var body: some View {
        Button(action: {
            showNoteSheet = true  // ← Simple toggle
        }) {
            // ... button content
        }
    }
    // .sheet modifier removed
}
```

**Result:** ✅ Sheet now presents correctly and stays open! The sheet state is managed at ContentView level, which doesn't re-evaluate when player state changes.

### ✅ T23: Home Screen Carousel Padding Regression Fix

**Commits:** `4cbc42d`, `8013695`

**Problem:**
The "Continue Listening" and "Podcasts" horizontal carousels on the home screen had their left and right sides cropped. Cards should run to the screen edge for a proper carousel effect, but the outer VStack's `.padding(.horizontal, EchoSpacing.homeSidePadding)` was clipping all content, including the horizontal ScrollViews.

**Root Cause:**
In HomeView.swift, the outer VStack (line 128) had `.padding(.horizontal, EchoSpacing.homeSidePadding)` applied. This padding clipped all child views, including the horizontal ScrollViews in "Continue Listening" and "Podcasts" sections, preventing cards from extending to the screen edges.

**Solution:**
Removed horizontal padding from the outer VStack and strategically re-applied padding:
1. Non-ScrollView content (header, "Recent Notes", "Empty State"): Added `.padding(.horizontal, EchoSpacing.homeSidePadding)` directly to these sections
2. Sections with horizontal ScrollViews ("Continue Listening", "Podcasts"): Applied padding ONLY to section headers, not the ScrollView wrapper
3. Horizontal ScrollViews: Added `.scrollClipDisabled()` and `.padding(.leading, EchoSpacing.homeSidePadding)` to HStacks inside ScrollViews

**Code Changes:**


## Key Learnings

1. **`.tabViewBottomAccessory` closure behavior:** The closure re-evaluates when @ObservedObject properties change, causing child views to reconstruct and lose local @State.

2. **Sheet presentation best practice:** Present sheets from the most stable view scope possible, especially when dealing with complex layouts like `tabViewBottomAccessory`.

3. **Dependency injection pattern:** Passing @ObservedObject as a dependency (not storing singleton reference) helps maintain view identity but isn't sufficient for tabViewBottomAccessory closure issues.

4. **State management:** Moving state up to a stable parent view (ContentView) is often the cleanest solution when child views are being reconstructed by container re-evaluation.
